01

What avc-parser Does

Trace a real operation end-to-end: from raw audit chaos to actionable denial groups

The Audit Log Problem

SELinux logs every policy denial to /var/log/audit/audit.log. A single misconfiguration can generate thousands of nearly-identical denials. Raw audit output looks like this:

audit.log type=AVC msg=audit(1234567890.001:101): avc: denied { read } for pid=1234 comm="httpd" path="/var/www/html/index.html" scontext=system_u:system_r:httpd_t:s0 tcontext=system_u:object_r:httpd_sys_content_t:s0 tclass=file type=AVC msg=audit(1234567890.002:102): avc: denied { open } for pid=1234 comm="httpd" path="/var/www/html/index.html" scontext=system_u:system_r:httpd_t:s0 tcontext=system_u:object_r:httpd_sys_content_t:s0 tclass=file ... 998 more similar denials

All 1,000 denials need the same fix. Reviewing them individually is a waste of time. That's the problem avc-parser solves.

The Solution: Intelligent Grouping

avc-parser transforms overwhelming audit logs into a manageable set of actionable denial groups. Each group represents a distinct policy issue that needs a single fix.

1
Validate and detect input format

Auto-detects raw audit.log vs. pre-processed ausearch output, validates file integrity

2
Parse and correlate audit records

Extracts AVC denials with full context from SYSCALL, PATH, CWD, PROCTITLE, and SOCKADDR records

3
Generate smart signatures and deduplicate

Groups denials by remediation pattern: 10,000 raw denials become 20 actionable groups

4
Format and display results

Outputs Rich terminal UI, JSON for automation, brief executive reports, or detailed sealert-style analysis

Architecture Overview

The cast of components and where each one lives

Project Structure

avc-parser follows a pipeline architecture with the orchestration logic in a single top-level module (parse_avc.py) and domain concerns split into focused packages.

avc-parser/
parse_avc.py ~4,800 lines. CLI, parsing, aggregation, display orchestration
config/ AUDIT_RECORD_RE regex, file size limit, format detection line count
validators/ File and argument validation with Rich error messages
avc_selinux/ AvcContext class, PermissionSemanticAnalyzer
detectors/ Anomaly detection: permissive mode, dontaudit, containers
formatters/ JSON, report (brief/sealert), stats output formatters
utils/ File detection, SELinux utils, sorting, time, legacy helpers
tests/ 174 tests: parsing, integration, edge cases, display, regression
docs/ CLI reference, examples, deduplication algorithm docs

Component Responsibilities

M
parse_avc.py (Orchestrator)

The main module. Houses CLI argument parsing, the full parsing pipeline, smart signature generation, deduplication logic, and Rich display output. Everything flows through here.

V
validators/

Validates CLI arguments and input files. Detects conflicting flags, checks file existence/permissions/size, and provides actionable Rich error messages with suggested fixes.

S
avc_selinux/

Domain model for SELinux. AvcContext parses user:role:type:mls context strings. PermissionSemanticAnalyzer maps permissions and object classes to human-readable descriptions.

D
detectors/

Anomaly detection functions that scan denial data for permissive mode, disabled dontaudit rules, container storage paths, and non-standard filesystem locations.

F
formatters/

Output formatters for JSON (SIEM integration), report format (brief executive and sealert technical), and stats summary. Each formatter takes the same denial data and produces different output.

The Parsing Pipeline

How raw audit text becomes structured denial data with full forensic context

Input Detection and Validation

The pipeline starts with detect_file_format() in utils/file_utils.py. It reads the first few lines looking for ausearch output markers.

CODE 
def detect_file_format(file_path):
    with open(file_path) as f:
        lines = [f.readline() for _ in range(10)]
    content = "\n".join(lines)

    has_time_headers = "time->" in content
    has_separators = "----" in content
    has_human_ts = re.search(
        r"\d{2}/\d{2}/\d{4} \d{2}:\d{2}:\d{2}",
        content)

    if has_time_headers or has_separators \
       or has_human_ts:
        return "processed"
    return "raw"
WHY IT WORKS THIS WAY

Only reads 10 lines because format detection doesn't need the full file. Fast O(1) check regardless of file size.

Checks for three independent markers because ausearch output varies across versions. Any single marker is sufficient evidence.

Human-readable timestamps (MM/DD/YYYY) only appear in ausearch -i output. Raw audit.log uses Unix epoch floats.

Returns "raw" by default because raw format requires running ausearch as a subprocess. Processed files can be read directly as text.

When the format is "raw", main() shells out to ausearch with specific message type filters to extract and correlate related records.

The Core Regex: AUDIT_RECORD_RE

Every audit record passes through a single regex pattern adapted from setroubleshoot. This is defined in config/constants.py:

config/constants.py AUDIT_RECORD_RE = re.compile( r"(node=(\S+)\s+)?" # Optional hostname prefix r"(type=(\S+)\s+)?" # Record type (AVC, SYSCALL, etc.) r"(msg=)?" # Optional msg= prefix r"audit\(" # Start of audit event ID r"((\d+)\.(\d+):(\d+))" # timestamp.ms:serial r"\)\s*:\s*" # End of event ID + colon r"(.*)" # Message body )
💡
Architectural Insight

The regex uses optional groups (...)? because audit records vary wildly in practice: some have node= prefixes, some omit type=, some add extra whitespace before the colon. A single flexible pattern handles all variants instead of maintaining multiple format-specific parsers.

Shared Context from Non-AVC Records

An audit log block can contain multiple record types sharing the same event ID. The function extract_shared_context_from_non_avc_records() pulls context from companion records and merges it into each AVC denial.

📂

SYSCALL

System call name, executable path, exit code, success/fail. Reveals what kernel operation triggered the denial.

📁

PATH

File paths, inode numbers, device identifiers. Multiple PATH records may exist, so the parser selects the best one using nametype priority.

📄

CWD

Current working directory of the process. Used to resolve relative paths found in PATH records to absolute paths.

💻

PROCTITLE

Full command line of the process. Falls back to comm field if PROCTITLE is null or missing.

🌐

SOCKADDR

Socket address information in hex format. Parsed for local address and port details on network-related denials.

After extracting shared context, process_individual_avc_record() merges it into each AVC line's parsed data, applies path normalization, resolves relative paths, and adds semantic analysis from PermissionSemanticAnalyzer.

AVC Context Parsing

SELinux contexts like system_u:system_r:httpd_t:s0 are parsed by the AvcContext class into structured components for analysis and display.

CODE 
class AvcContext:
    def __init__(self, context_string):
        self.user = None
        self.role = None
        self.type = None
        self.mls = None

        if isinstance(context_string, str) \
           and context_string:
            fields = context_string.split(":")
            if len(fields) >= 3:
                self.user = fields[0]
                self.role = fields[1]
                self.type = fields[2]
                if len(fields) > 3:
                    self.mls = ":".join(
                        fields[3:])
                else:
                    self.mls = "s0"
WHY IT WORKS THIS WAY

All four fields start as None so downstream code can check for incomplete contexts with is_valid() without raising exceptions.

Type-checks and emptiness-checks before splitting because real audit logs can contain garbled or empty context strings.

Requires at least 3 colon-separated fields (user:role:type) since that's the minimum valid SELinux context.

MLS labels can contain colons (e.g., s0:c0.c1023), so everything after the third colon is joined back together rather than taking just fields[3].

Defaults to "s0" when MLS is missing. This is a known trade-off documented with a TODO for future improvement.

04

Smart Deduplication

The core innovation: grouping denials by remediation pattern, not exact match

Remediation-Aware Grouping

Traditional deduplication compares exact fields: same source, target, class, and permission. That still leaves hundreds of "unique" denials because different files, PIDs, and timestamps make each record technically distinct.

The key insight: denials that need the same policy fix should be in the same group. The smart signature captures this by categorizing each field into its remediation bucket.

💡
Architectural Insight

The smart signature is a composite key stored as a Python tuple. Using tuples as dictionary keys gives O(1) lookup for grouping, so deduplication scales linearly with input size regardless of the number of groups.

The Five Signature Components

process_category Groups related process variants: httpd, httpd-worker, httpd-prefork all become web_server_apache
target_context The full target SELinux context, unchanged. This is the "what are we accessing?" and directly maps to semanage fcontext rules.
object_group Groups related object classes: file, dir, lnk_file become filesystem. Policy rules often apply to all filesystem types together.
path_pattern Normalizes paths to parent directories: /var/www/html/index.html and /var/www/html/style.css both become /var/www/html/
permission_category Groups permissions by operation type: read, open, getattr become read_operations. All need the same allow rule.

Inside generate_smart_signature()

CODE 
def generate_smart_signature(
        parsed_log, legacy_mode=False):
    scontext = parsed_log.get("scontext")
    tcontext = parsed_log.get("tcontext")
    tclass = parsed_log.get("tclass", "")
    perm = parsed_log.get("permission", "")
    comm = parsed_log.get("comm", "")
    path = parsed_log.get("path", "")

    if legacy_mode:
        return (context_to_str(scontext),
                context_to_str(tcontext),
                tclass, perm)

    process_cat = get_process_category(
        comm, scontext)
    perm_cat = get_permission_category(
        perm, tclass)
    obj_group = get_object_group(tclass)
    path_pat = get_path_pattern(
        path, tclass, parsed_log)

    return (process_cat,
            context_to_str(tcontext),
            obj_group, path_pat, perm_cat)
WHY IT WORKS THIS WAY

Extracts the five raw fields needed for categorization. All use .get() with defaults because corrupted records may be missing fields.

Legacy mode exists for regression testing. It returns the old exact-match 4-tuple so you can compare grouping behavior between algorithms on the same dataset.

Each categorizer function is pure: takes raw values, returns a category string. This makes them independently testable and composable.

Target context is kept unchanged because it directly maps to semanage fcontext rules. Categorizing it would lose remediation specificity.

The 5-tuple is used as a dict key for O(1) grouping. All denials producing the same tuple end up in the same group.

Output and Formatting

Five output strategies from a single denial data structure

Output Modes

All output modes consume the same unique_denials dictionary. The CLI flag determines which formatter runs, following a strategy pattern.

📊

Rich Summary (default)

Full Rich terminal UI with BIONIC text formatting, correlation trees, resource grouping, and anomaly indicators. Lives in parse_avc.py.

{}

JSON (--json)

Normalized structured output for SIEM systems, AI tools, and automation. Includes sesearch commands and field normalization. In formatters/json_formatter.py.

📝

Brief (--report brief)

Executive summary with WHAT/WHEN/WHO/WHERE/IMPACT/STATUS format. For management reporting and compliance documentation.

🔍

Sealert (--report sealert)

Technical analysis format inspired by setroubleshoot. Includes raw audit reconstruction, analysis details, and event distribution per PID.

#

Stats (--stats)

Aggregate statistics: top source types, target types, object classes, permissions, and processes. Quick overview without individual denials.

Semantic Enrichment

Before formatting, each denial is enriched with human-readable descriptions by PermissionSemanticAnalyzer. This class maps raw SELinux identifiers to contextual explanations.

CODE 
@classmethod
def get_contextual_analysis(
        cls, permission, obj_class,
        source_context=None,
        target_context=None,
        process_name=None):

    source_desc = "Process"
    if process_name:
        source_desc = process_name
    elif source_context:
        source_desc = source_context\
            .get_type_description()

    context_patterns = {
        ("read", "file"):
            f"{source_desc} attempting "
            f"to read file content",
        ("name_connect", "tcp_socket"):
            f"{source_desc} attempting "
            f"to connect to network",
        # ... 20+ patterns
    }

    key = (permission, obj_class)
    if key in context_patterns:
        return context_patterns[key]
    return f"{source_desc} attempting "
           f"{perm_desc} on {target}"
WHY IT WORKS THIS WAY

Prioritizes the actual process name (comm field) over the SELinux type description because "httpd" is more meaningful to a sysadmin than "Web server process".

Uses a static dictionary keyed by (permission, object_class) tuples. This avoids requiring access to the actual SELinux policy files, making the tool portable across systems.

Falls back to a generic description template when no specific pattern matches. This ensures every denial gets some human-readable context, even unusual permission/class combinations.

The @classmethod design means no instance state: every call is self-contained, making it safe to use from any thread or context without initialization.

Anomaly Detection

Before output, the detectors/ package scans denial data for conditions worth highlighting. Each detector is a pure function that returns a result tuple.

detect_permissive_mode() Flags denials where permissive=1. These were logged but allowed, meaning policy enforcement is weakened.
detect_dontaudit_disabled() Looks for permissions like noatsecure, rlimitinh, siginh that are normally suppressed. Their presence means semodule -DB was run.
has_container_paths() Checks for Podman/Docker overlay storage paths. Container denials often need different remediation than host-level denials.
has_custom_paths() Flags non-standard paths like /opt, /app, /data. These usually indicate custom software that needs custom policy rules.
💡
Architectural Insight

Every detector is a pure function: takes a denial dict, returns a result. No side effects, no state. This makes them trivially testable and composable. Adding a new detector means writing one function and calling it from print_rich_summary().

Extension Points

How to add new functionality while following the project's conventions

Adding a New Output Formatter

Each formatter is an independent function that takes the unique_denials dictionary and writes output. Follow this pattern:

1
Create formatters/yaml_formatter.py

Define a function like format_as_yaml(unique_denials, valid_blocks, generate_sesearch_command) matching the same signature as format_as_json().

2
Export from formatters/__init__.py

Add from .yaml_formatter import format_as_yaml to the package init so it's importable from the top level.

3
Add CLI flag in main()

Add --yaml argument to the argparse setup and wire it into the output selection block alongside --json.

4
Add tests in tests/

Follow the pattern in test_integration.py which uses subprocess to run the CLI and validates output structure.

Adding a New Anomaly Detector

Detectors are pure functions. Here's the template:

detectors/anomaly_detector.py def has_high_frequency_denials( denial_info: dict ) -> tuple[bool, int]: """ Check if a denial group has unusually high event frequency. """ count = denial_info.get("count", 0) first = denial_info.get("first_seen_obj") last = denial_info.get("last_seen_obj") if first and last and count > 1: duration = (last - first).total_seconds() if duration > 0: rate = count / duration return rate > 10, int(rate) return False, 0

Then call it from print_rich_summary() alongside the existing detector calls and add an indicator badge to the header.

Testing Conventions

The project uses pytest with unittest-style test classes. Key patterns to follow:

🧪

Unit Tests

Import functions directly and test with crafted input strings. See test_core_parsing.py for AvcContext, regex, and field extraction tests.

🔄

Integration Tests

Use subprocess.run() to invoke the CLI and validate exit codes, JSON structure, and error messages. See test_integration.py.

⚠️

Edge Case Tests

Test with malformed input, empty strings, missing fields, and corrupted records. See test_edge_cases.py and test_malformed_logs.py.

🔀

Regression Tests

Verify specific bug fixes stay fixed. test_regression.py covers sorting, filtering, quoting, and USER_AVC edge cases.
